# Provenance

This repository builds `a-op` from Michael Wall's privately owned Sound for Movement web platform and years of operating an independent music company.

## Authorized visual foundation

On 2026-07-18, Michael explicitly authorized the live Sound for Movement visual framework as the starting visual framework for `a-op`. This authorization covers the design system, semantic color values, Lato typography, spacing, corners, surfaces, controls, light and dark themes, motion, responsive behavior, accessibility patterns, and overall page composition. Michael subsequently removed the living mosaic from the `a-op` baseline and directed the neutral installation to carry no temporary image assets.

The React implementation reproduces that framework with neutral `a-op` content and general product names. `Courses` replaces the Sound for Movement teaching-area name. `What's New` remains the in-app update name. `docs/architecture/visual-direction.md` records the authorized system.

## Private material

The private Sound for Movement repository remains separate and read-only. Its machine-specific path belongs only in ignored local configuration. Sound for Movement retains:

- Sound for Movement branding, logos, public copy, private documents, or production URLs.
- Michael's music, artwork, photographs, video, customer records, analytics, or business secrets.
- Environment variables, tokens, keys, webhook secrets, connection strings, or local filesystem paths.
- Historical migrations or business-specific rules that do not express the reusable platform contract.

## Functional reference

Reusable concepts may be studied and implemented in the Sites application: music catalog relationships, audio playback behavior, server-side authorization, entitlements, memberships, subscriptions, licensing workflow, Courses, video, administration, telemetry, testing, and operational recovery.

Each new Sites implementation entry must name the concept, private reference area when safe, new `a-op` files, the implementation decision, the relevant commit, and verification. Private content stays in its existing private location.

The existing ledger records the completed Nuxt implementation. Its earlier visual entries describe the decision in effect when those commits were created. The current Sites decision begins from the exact authorized visual foundation above.
